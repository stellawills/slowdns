# Security Policy

## Supported Scope

Security reports should cover the current `main` branch and the production
install flow documented in [README.md](README.md).

## Reporting A Vulnerability

Do not open public GitHub issues for security vulnerabilities.

Report security issues privately with:

- a short summary of the issue
- affected file paths or commands
- reproduction steps
- impact
- suggested fix if you already have one

Use the project's private security contact or maintainer-operated security
channel.

If you are not sure whether something is security-sensitive, report it
privately first.

## Response Expectations

- Initial triage target: within 3 business days
- Status update target: within 7 business days
- Fix timing depends on severity and operational risk

## Out Of Scope

The following are usually not treated as reportable vulnerabilities on their
own:

- missing best-practice headers with no exploit path
- self-hosted test environment issues
- rate-limit bypass claims without a working abuse path
- theoretical supply-chain concerns without a concrete trust break
